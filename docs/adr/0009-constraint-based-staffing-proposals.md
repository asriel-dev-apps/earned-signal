# ADR 0009: Constraint-based Staffing Proposals

## Status

Accepted

## Context

Staffing changes can affect dates, planned labor cost, overload, and Skill coverage, but Current must not change until a human approves an exact diff. The present task model has duration and Assignment demand, but no authoritative remaining-work quantity. Treating Assignment demand or budget as productivity would silently invent effort. CP-SAT is native compute and does not belong in the Workers JavaScript request path. An LLM can make a verified result easier to understand, but must not calculate or alter the plan.

## Decision

- A Staffing Proposal is a separate, tenant-scoped aggregate. It stores a revision-pinned immutable input and canonical hash, lifecycle status, actor, and audit history. Its append-only terminal Run stores the Application result, algorithm version, verified metrics and changes when feasible, diagnostics, explanation, and solver metadata (OR-Tools version, deterministic seed, worker count, per-stage limits, and completed objective values/bounds).
- Every unfinished task included in a solve requires positive integer `remainingEffortMinutes` with `HUMAN_CONFIRMED` provenance. A UI may show a duration-based suggestion, but the request records only the value a human submitted. Missing effort fails preflight as `MISSING_REMAINING_EFFORT`.
- Initial solutions use existing Resources plus explicitly supplied candidate Resources. Assignment units are constant for a task and come from the finite set `25, 50, 75, 100`. Each task also has an explicit maximum parallel Resource count. Daily or preemptive Assignments are not introduced.
- Hard constraints cover target finish, planned labor cost, total overtime minutes, Skill coverage, maximum changed Task/Resource Assignment pairs, maximum changed Task schedules, and maximum candidate Resources. One Assignment-pair change means that a pair is added, removed, or has a different units percentage from Current; it is not the number of `assignment.replace` commands. Planned labor cost uses the existing Resource rate without an overtime premium because no premium rate is modeled. Total overtime is the sum, across all Resource/date rows, of aggregate assigned demand above available calendar capacity. The initial contract has no separate per-Resource or per-day overtime ceiling.
- Feasible plans are compared in one required lexicographic order: earliest finish, least total overtime, lowest planned labor cost, then fewest changed Assignment pairs. Further stability stages minimize changed Task schedules, candidate Resources, an ID-sorted weighted sum of Assignment units, and an ID-sorted weighted sum of start-day indices when not already covered. Weighted sums can collide and therefore do not prove plan uniqueness. Clients and LLMs cannot reorder the objectives or supply expressions or weights.
- The Python OR-Tools service uses integer CP-SAT variables, seed `20260716`, one search worker, bounded model size, a five-second wall-clock limit per stage, and deterministic-time limit `1.0` per stage. Every response reports these values, the solver version, and completed objective values and best bounds. `OPTIMAL`, `FEASIBLE`, `INFEASIBLE`, `UNKNOWN`, and `MODEL_INVALID` remain distinct. A time-limited feasible result is never described as optimal.
- Request constraint families that participate in assumption-based diagnostics include date constraints, deadline, total overtime, cost, Assignment-pair changes, schedule changes, candidate use, and required Skills. Effort, availability, dependency, and duration structure remain structural model constraints. An infeasible response may report a sufficient, not necessarily minimal, assumption core; it must not call that set the unique cause. The current implementation does not calculate single-constraint relaxations.
- Solver output is untrusted. The Application layer checks the source revision and solver metadata, converts the result only to plan-safe `resource.add`, `assignment.replace`, and `task.update` commands, and reapplies them with the normal command rules. It independently recomputes the canonical schedule, confirmed-effort coverage, capacity, total overtime, planned labor cost, Assignment-pair changes, Task schedule changes, candidate count, and Skill coverage. Solver-reported aggregate totals do not substitute for this calculation; any mismatch with a hard limit rejects the result.
- READY completion appends the Proposal Run and creates and links a DRAFT Scenario with the exact verified commands in one PostgreSQL transaction at the same still-current Project revision. A generic terminal write cannot produce READY. The Workflow calculates and persists the deterministic Scenario Run in a subsequent retryable step, so the Run is not part of the READY-link transaction. Scenario status is the source of truth for approval or rejection. Existing human-only, stale-safe, atomic Scenario publication is the only path to Current; Baseline never changes.
- Proposal generation is asynchronous. REST and MCP perform the same request validation, authorization, idempotent PostgreSQL creation, and Workflow dispatch; neither transport runs or accepts a solve. The Web Worker commits the Proposal before starting or observing a Cloudflare Workflow whose instance ID is the Proposal ID. A dedicated optimizer Worker calls a private, network-disabled, per-Proposal Cloudflare Container containing pinned Python and OR-Tools. PostgreSQL is the system of record; Workflow state and Container disk are not.
- Transport and host failures in the solve-and-verify Workflow step are retryable. Solver statuses are calculation results and are not retried. The Proposal ID is also the Workflow instance ID; the canonical immutable request hash, first-terminal-result rule, and conditional terminal writes prevent a replay or late result from replacing accepted state.
- Queue is not part of the initial path. It may later dispatch Workflow creation if measured burst or backpressure requires it; a Queue consumer must never perform the solve itself.
- Workers AI receives only verified fact strings and the exact serialized command diff after Application revalidation. It cannot change numbers, commands, constraints, solver status, or approval state. Bounded output containing numeric, date, or identifier-like claim tokens not recognized in those inputs is rejected, and a deterministic factual explanation is used when AI is unavailable or its output is invalid.

## Consequences

Proposal creation can be granted to an Agent with a dedicated propose scope without granting Scenario mutation or publication. Human users can inspect the exact solver diff and Application-verified forecast before approval. The stored input, Proposal Run algorithm version, solver metadata, objective values/bounds, diagnostics, and output support replay analysis; the service image digest is deployment evidence and is not currently stored in the Proposal Run. An `OPTIMAL` result proves the recorded objective and stability-stage values, but the weighted stability stages do not prove a unique selected plan. The intended beta topology uses Cloudflare Containers on the Workers Paid plan; production memory, cold starts, host restarts, and representative solve sizes must be measured before a commercial SLO is claimed. Monte Carlo forecasting is implemented separately under [ADR 0010](0010-reproducible-monte-carlo-forecasting.md) and does not change this staffing decision.
