# ADR 0010: Reproducible Monte Carlo Scenario forecasting

## Status

Accepted for the beta Scenario workflow.

## Context

A deterministic trend is useful for inspecting a plan change, but it cannot answer how likely a finish date or cost outcome is when remaining effort is uncertain. The probabilistic result must be reproducible, must not overwrite the deterministic Scenario Run used by the publication guard, and must remain safe when Cloudflare Queues redelivers a message.

## Decision

- `forecast.v1` accepts exactly one optimistic, most-likely, and pessimistic remaining-effort estimate in whole minutes for every unfinished Task. Each estimate is explicitly marked `HUMAN_CONFIRMED`; completed Tasks and inferred estimates are rejected. The simulator samples the corresponding triangular distribution.
- An optional correlation group associates two or more unfinished Tasks with one coefficient from 0 to 9,500 basis points. A Task belongs to at most one group. The simulator combines one group-normal and one Task-normal factor, then maps their normal CDF through the triangular quantile. This models positive within-group effort correlation without accepting an arbitrary covariance matrix.
- The caller supplies an unsigned 32-bit seed. The exact canonical-input hash, algorithm version, PRNG and normal-source method, distribution method, schedule method, runtime version, seed, actual iteration count, and every stopping checkpoint are returned as replay evidence.
- The stopping rule permits 1,000 to 50,000 iterations. It checks ordered P50/P80 schedule and cost quantiles at a configured interval and stops only after the requested number of consecutive relative-tolerance checks, or at the maximum. Identical canonical input, seed, algorithm/runtime version, and platform produce identical output.
- Application applies Scenario plan changes through the normal command validation and calculates the deterministic Scenario schedule before building the simulation problem. It includes only unfinished Tasks and converts the Project/Resource calendars into a bounded 366-day working-date horizon. Each Task starts no earlier than both the status-date boundary and its scheduled early start, preserving future plan constraints. A Task's productive minutes and weighted hourly rate come from its complete Assignment set. Working dates are the conservative intersection of the default Project, Task, and all assigned Resource calendars, so the constant daily productive capacity is valid on every supplied date.
- Dependencies retain their type and non-negative working-day lag. Dependencies on completed predecessors are omitted because their effect is already represented at the status-date boundary. Actual cost is copied per unfinished Task, completed-Task actual cost is supplied once at the Project level, and sampled remaining labor cost is added separately; completed cost is therefore retained without being repeated in every simulation Task.
- The untrusted simulator response is accepted only after Application verifies the canonical-input hash, Project and source revision, seed metadata, iteration bounds, checkpoint spacing and consecutive stopping semantics, P50/P80 ordering, exact histogram-derived quantiles and target probability, histogram ordering and counts, safe monetary ranges, and exact versioned algorithm metadata.
- A Forecast Run is a separate tenant/Project/Scenario-scoped aggregate pinned to exact Project and Scenario revisions. Its immutable input and canonical SHA-256 request hash are stored once, sent to the simulator, independently recomputed there, returned, and compared before persistence. Lifecycle transitions are normally `REQUESTED → RUNNING → READY|FAILED`; a stale, permanently invalid, or retry-exhausted request may move directly from `REQUESTED` to `FAILED`. The first terminal result wins. Results and audit events are append-only.
- Forecast creation is idempotent within a Scenario. The Queue consumer explicitly acknowledges each successful or permanently rejected message and retries transient failures with bounded backoff. On the sixth delivery it records `FORECAST_RETRIES_EXHAUSTED`; if that terminal write is unavailable, the configured dead-letter Queue is consumed by the same idempotent terminalizer with an extended retry window. Queue start and completion remain replay-safe, including late delivery after a terminal Run. A non-terminal Run fails closed when the Project or Scenario revision changes. Forecast results never replace `scenario_runs.latest_run_id` and never authorize Scenario publication.

## Consequences

The P50/P80 finish, P50/P80 total cost, target-date probability, and histograms can be explained and replayed from a stored input. The fixed one-group-per-Task model intentionally does not represent negative correlation or overlapping systemic risks. The conservative common-working-date calendar can overstate duration when partially available teams could continue work; a future contract version may represent date-varying capacity without changing stored `forecast.v1` meaning. The 366-day horizon, 100-Task and 50,000-iteration limits bound request size and compute for the beta. Topological order and calendar lookup indexes are prepared once outside the sampling loop; the repository includes a reproducible maximum-bound benchmark rather than treating the numeric limits alone as a performance guarantee.
