# Application context

The Application context is the common mutation and use-case boundary for Web, REST, and MCP. Commands validate project invariants before returning a new immutable state. Adapter-specific validation must not replace these guarantees.

Current commands add, update, and delete leaf work packages and Resources, atomically replace all Assignments for one work package, and publish Current as a named immutable Baseline version. Project state validates WBS parent membership and cycles, calendar references, multiple typed dependencies, scheduling constraints, Skills, Resources, and Assignments. ProjectCommandService is the shared Web, REST, and MCP use-case boundary. Its ProjectCommandUnitOfWork port surrounds validation with persistence, canonical idempotency, optimistic concurrency, and auditing; adapters must not duplicate those rules.

ProjectCommandAuthorizer is the shared authorization boundary for Current mutations. Human project owners/editors may mutate Current; viewers may not. Agent identities may directly write only progress and actuals when the required scope exists in both their stored grant and signed access token. Agent plan commands are never applied directly. A separate StaffingProposalAuthorizer allows a scoped Agent to request a revision-pinned Proposal, but only a human owner/editor may publish its linked Scenario into Current. A successful authorization returns the stable internal principal ID used as the AuditActor.

ProjectQueryAuthorizer permits a provisioned owner, editor, viewer, or agent to read its project while rejecting identities without the tenant/project grant. Query adapters must use it before loading project performance or other project data.

Scenario calculation applies an ordered, plan-only subset of Project Commands to an immutable Current input. It reuses Project validation, scheduling, and capacity calculation, then applies deterministic SPI/CPI factors only to unfinished duration and remaining budget. Scenario commands cannot publish a Baseline or write progress and actuals.

Staffing Proposal generation sends a versioned, immutable Current snapshot and human-confirmed remaining effort to a numeric optimizer. The objective order is fixed as earliest finish, least total overtime, lowest planned labor cost, then fewest changed Task/Resource Assignment pairs. Clients cannot reorder it or supply weights. The accepted overtime bound and metric are the total overallocated minutes across every Resource/date row in the complete proposed Project plan, including unchanged completed-Task Assignments; the Application layer does not use the solver's unfinished-Task total as authority. The Assignment-change bound and metric count pairs whose presence or units differ from Current, not the number of `assignment.replace` commands.

Solver output can only become Scenario plan commands after the Application layer checks the source revision and solver metadata, permits only selected candidate Resource additions, complete Assignment replacements, durations, and optional starts, and reapplies every command through normal Project validation. It then recomputes the authoritative schedule, human-confirmed effort coverage, Resource capacity, total overtime, planned labor cost, Assignment-pair changes, Task schedule changes, candidate usage, and Skill coverage. Solver-reported totals are not the accepted numeric source of truth. Stored solver metadata includes the solver version, deterministic seed, one-worker setting, wall-clock and deterministic limits per stage, and each completed objective's value and best bound; an `OPTIMAL` result is rejected unless its returned stages are bound-proven. The explanation adapter receives only verified facts and exact changes, and invalid or unavailable prose falls back to a deterministic explanation.

Probabilistic Forecasts use the separate `forecast.v1` boundary. Application first validates and applies the Scenario plan, requires a human-confirmed triangular remaining-effort estimate for every unfinished Task, validates disjoint positive-correlation groups, a uint32 seed, bounded stopping rules, and the target date, then derives a revision-pinned Python problem. The adapter supplies working dates where the Project, Task, and every assigned Resource calendar overlap, constant productive minutes and capacity-weighted cost rate, typed dependencies, and actual cost. Untrusted output is accepted only after revision, seed/version metadata, stopping, quantile, probability, histogram, date, and money checks succeed.

## Language

- **Owner**: the accountable label on a work package; it does not reserve capacity.
- **Assignment replacement**: one command that supplies the complete desired Assignment set for a work package, including an empty set to clear it.
- **Plan change**: a Current mutation other than progress or actual recording; Resource and Assignment commands are plan changes.
- **Project query**: a read-only project operation authorized by tenant/project membership rather than mutation role or agent write scopes.
- **Baseline publish**: a human plan command that freezes the complete Current plan as the next approved, immutable Baseline version.
- **Scenario plan command**: a Task, Resource, or Assignment plan command evaluated in order against an in-memory Scenario without mutating Current.
- **Human-confirmed remaining effort**: a positive whole-minute forecast and a positive maximum parallel-Resource count supplied for every unfinished Task. They are optimizer inputs, not inferred from duration or progress, and cannot be changed by solver output.
- **Staffing Proposal**: a revision-bound set of explicit candidate-Resource, Assignment, duration, and start-date changes that passed deterministic Application validation and still requires Scenario publication by a human.
- **Assignment-pair change**: one Task/Resource pair added, removed, or changed to a different units percentage relative to Current. One `assignment.replace` command may contain zero, one, or multiple Assignment-pair changes.
- **Total overtime**: the sum of demand above available capacity for every Resource/date row in the proposed plan, expressed in minutes.
- **Forecast Run**: an asynchronous, immutable Monte Carlo request and first terminal result pinned to one Project and Scenario revision; it never serves as the deterministic Run required for publication.
- **Human-confirmed three-point estimate**: optimistic, most-likely, and pessimistic remaining effort in whole minutes for one unfinished Task, ordered from low to high and explicitly confirmed by a human.
